How the mapping works

Fields used directly as targets

Several fields are collected in the same shape they're used for stratification, so they flow straight across with no transformation:

  • gender → gender
  • disability → disability
  • education_level → education_level
  • occupation → occupation

Even "direct" fields still come from a fixed set of options agreed with the client, so the recorded value already matches a target category.

Date of birth → age → age_bracket

The three separate fields dob_day, dob_month and dob_year are combined to calculate an intermediate value, age. The age is then mapped into an age_bracket, and it's the bracket — not the raw age — that is used as the target.

The brackets used in this example, with a note on some special cases:

BracketNote
0–17Target is zero — under-18s are not selected
18–24
25–34
35–50
51–64
65–74
75–100
100+Target is zero; any 100+ value is double-checked to confirm the data is actually valid

So the flow is: dob_day + dob_month + dob_year → age → age_bracket.

Ethnicity → ethnicity_bucket

The form offers 10 ethnicity options, deliberately fine-grained so that people feel included and can see themselves represented. For the target, though, those options are combined into a smaller number of ethnicity_bucket values.

For example, several "White" options collapse into a single bucket:

  • White British, White Irish, White - other → White

So the field value is recorded at full granularity, and the bucketing happens on the way to the target.

Post code → geo_bucket and rural_urban

The single post_code field feeds two separate targets, each derived through its own lookup table:

  • post_code → lookup table → geo_bucket
  • post_code → lookup table → rural_urban

The registrant only ever enters their postcode; the geographic bucket and the rural/urban classification are looked up from it, so neither is asked for directly.

Address fields → address_check (the exclusion mechanism)

Invitation letters were posted to a randomly selected set of addresses. When someone registers, the address they enter — address_line_1, address_line_2, city and post_code — is matched against that list of addresses we wrote to:

  • Address matches one we posted a letter to → address_check = OK
  • No match → address_check = BAD

address_check is then used as a target with a value of zero for BAD. Because the selection cannot pick anyone into a category whose target is zero, this is the mechanism that excludes registrations from addresses we didn't invite.

You can read this either way round: as a target (shown on the right of the diagram), or as a post-submission filter that drops invalid registrations. They're the same thing — the zero target is the drop.

One field, used more than once

A field appears exactly once on the registration page, but the same value can be used in several ways. post_code is the clearest example:

  • As part of the registrant's contact address
  • Derived into the geo_bucket target
  • Derived into the rural_urban target
  • Part of the address_check match (with the other address fields)

There may be others — the point is that "collected once" does not mean "used once".

Fields that are not targets

Not every field feeds the stratified selection. In this example:

  • Must-check boxes — can_attend, eligible and consent all have to be ticked before the form can be submitted at all. They gate submission rather than describing the person.
  • Real choice — stay_on_db ("keep my details and contact me about similar events in the future") can be ticked or not; it drives a future-contact opt-in list, not selection.
  • Contact details — name, email, phone and address are used to reach the registrant and confirm attendance. (post_code is the exception that also feeds targets — see above.)
